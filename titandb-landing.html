🎯 Design Decisions

Decision 1: Single-Page B+ Tree (v1.0)

Choice: Implement single-page version first, defer multi-level to v2.0

✅ Advantages:
  • Simple implementation - easier to understand
  • Fast development iteration (12 hours total)
  • Easy to test and debug
  • Clear logic flow for learning
❌ Limitations:
  • Limited to 150 records per page
  • Can't scale to millions of records
  • No range scans yet
  • Single-page bottleneck

Decision 2: MVCC Over Locking

Choice: Multi-Version Concurrency Control for true parallelism

Aspect Locks (Traditional) MVCC (TitanDB)
Reader blocks writer? ❌ YES ✅ NO
Writer blocks reader? ❌ YES ✅ NO
Throughput Lower (blocking) Higher (parallel)
Complexity Simple Moderate
Memory Usage Lower Higher (versions)

Decision 3: Write-Ahead Logging (WAL)

Choice: Log to disk BEFORE applying changes to memory

WAL Process Comparison

❌ WITHOUT WAL
1. User calls insert()
2. Write to memory
3. CRASH! 💥
4. Data LOST! ❌
✅ WITH WAL
1. User calls insert()
2. Write to log on disk
3. fsync() - force flush
4. Update memory
5. CRASH! 💥 (log saved)
6. Replay log on restart ✅

Decision 4: LRU Buffer Pool

Choice: Least Recently Used caching strategy

💻 Quick Start

Install & Build

git clone https://github.com/yourusername/TitanDB.git
cd TitanDB
mvn clean test -Djacoco.skip=true

Basic Usage

ConcurrentDiskBPlusTree<Integer, String> db =
  new ConcurrentDiskBPlusTree<>("titandb.db", 4);

var txn = db.begin();
db.insert(txn, 1, "Alice");
db.commit(txn);

String value = db.search(txn, 1); // "Alice"

Concurrent Transactions (NO LOCKS!)

// Transaction 1: Writer
var txn1 = db.begin();
db.insert(txn1, 100, "Data");
db.commit(txn1);

// Transaction 2: Reader (concurrent!)
var txn2 = db.begin();
String value = db.search(txn2, 100); // "Data"
db.commit(txn2); // No blocking! MVCC! ✨
